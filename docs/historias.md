| ID   | Historia | Criterios de Aceptación | Prioridad | Story Points |
|------|----------|-------------------------|:---------:|:------------:|
| HU-01 | Como **cajero**, quiero registrar un cliente natural con sus datos personales para aperturar una cuenta en el sistema | - El sistema valida que el documento de identidad no esté repetido<br>- Valida que el email contenga @ y sea válido<br>- La edad del cliente es calculada correctamente a partir de fecha de nacimiento<br>- Solo se aceptan tipos de documento válidos (CÉDULA, PASAPORTE, NIT, CÉDULA_EXTRANJERÍA)<br>- El registro es exitoso solo con todos los datos requeridos completos | Alta | 3 |
| HU-02 | Como **cajero**, quiero registrar un cliente empresarial con sus datos y representante legal para disponer de cuentas comerciales | - El NIT debe ser único en el sistema<br>- Se valida que el representante legal tenga datos válidos<br>- Se valida que la razón social no esté vacía<br>- El registro falla si falta algún dato obligatorio<br>- Se almacena correctamente el cliente en la estructura del banco | Alta | 3 |
| HU-03 | Como **cajero**, quiero abrir cuentas (corriente, ahorros, crédito) para clientes ya registrados en el sistema | - No se pueden abrir más de 5 cuentas por cliente<br>- Lanza `CapacidadExcedidaException` si se intenta exceder el límite<br>- El número de cuenta es único<br>- La cuenta se crea con saldo inicial según el tipo<br>- Se registra la fecha de creación y el usuario que la abrió | Alta | 3 |
| HU-04 | Como **cajero**, quiero procesar depósitos en las cuentas de los clientes para que el saldo de la cuenta se actualice correctamente y la transacción quede registrada | - Solo permite depósitos en cuentas activas (no bloqueadas)<br>- Lanza `CuentaBloqueadaException` si la cuenta está bloqueada<br>- El monto debe ser mayor a cero, si no lanza `DatoInvalidoException`<br>- El saldo aumenta exactamente por el monto depositado<br>- Se registra la transacción con estado "COMPLETADA"<br>- Si el cliente acepta notificaciones, recibe mensaje de confirmación | Alta | 2 |
| HU-05 | Como **cajero**, quiero procesar retiros validando saldos y límites de retiro diario para evitar sobregiros no autorizados | - No permite retiros en cuentas bloqueadas (lanza `CuentaBloqueadaException`)<br>- No permite retirar más del saldo disponible (lanza `SaldoInsuficienteException`)<br>- Respeta el límite de retiro diario según el tipo de cuenta<br>- El saldo disminuye exactamente por el monto retirado<br>- Se aplica la comisión correspondiente según el tipo de cuenta<br>- Se registra la transacción con estado "COMPLETADA" | Alta | 3 |
| HU-06 | Como **asesor financiero**, quiero realizar transferencias entre cuentas del banco para que el dinero se mueva de forma segura entre clientes | - Solo permite transferencias entre cuentas activas y no bloqueadas<br>- Valida que exista saldo suficiente en la cuenta origen<br>- Aplica las comisiones correspondientes según tipo de cuenta origen<br>- Registra dos transacciones: una en la cuenta origen (EGRESO) y otra en destino (INGRESO)<br>- Ambas transacciones quedan con estado "COMPLETADA"<br>- Si falla, no se modifica ningún saldo | Media | 3 |
| HU-07 | Como **asesor financiero**, quiero consultar clientes y visualizar sus datos personales, cuentas asociadas e historial de transacciones | - Permite buscar cliente por documento o ID<br>- Lanza `ClienteNoEncontradoException` si el cliente no existe<br>- Muestra información personal (nombre, documento, email)<br>- Muestra todas las cuentas asociadas (máximo 5)<br>- Muestra el historial de transacciones de cada cuenta (máximo 20)<br>- Los datos se presentan de forma legible y completa | Media | 2 |
| HU-08 | Como **asesor financiero**, quiero bloquear y desbloquear cuentas según criterios de seguridad para proteger las cuentas de clientes | - Solo permite cambiar el estado de bloqueo de cuentas activas<br>- Valida que la transición de estado sea válida<br>- Se registra quién realizó el cambio y cuándo en la auditoría<br>- Las cuentas bloqueadas no permiten depósitos ni retiros<br>- Se notifica al cliente si tiene notificaciones activas | Media | 2 |
| HU-09 | Como **asesor financiero**, quiero cambiar el estado de una transacción (PENDIENTE → PROCESANDO → COMPLETADA) respetando las transiciones válidas | - Solo permite transiciones de estado válidas según el diagrama de transiciones<br>- Lanza `EstadoTransaccionInvalidoException` si la transición no es válida<br>- Se registra quién realizó el cambio en la auditoría<br>- Estados finales (RECHAZADA, REVERTIDA) no pueden cambiar a otros estados<br>- Cada cambio de estado queda registrado con fecha y usuario | Media | 2 |
| HU-10 | Como **gerente de sucursal**, quiero aprobar créditos para clientes que lo solicitan | - Solo el gerente de sucursal puede aprobar créditos (lanza `PermisoInsuficienteException` si otro tipo de empleado lo intenta)<br>- Se valida que el cliente existe y tiene buen historial<br>- Se registra el monto de crédito aprobado y el límite<br>- Se notifica al cliente de la aprobación<br>- La operación queda auditada con fecha, usuario y detalles | Alta | 3 |
| HU-11 | Como **gerente de sucursal**, quiero calcular la nómina total de todos los empleados de mi sucursal | - Calcula el salario de cada empleado según su tipo (Cajero, Asesor, Gerente)<br>- Incluye salario base + bonificaciones por antigüedad, desempeño, transacciones, etc.<br>- Solo suma empleados activos<br>- Muestra el total de nómina y desglose por tipo de empleado<br>- El cálculo es exacto y se puede auditar | Media | 2 |
| HU-12 | Como **gerente de sucursal**, quiero registrar nuevos empleados (cajeros, asesores, gerentes) en el banco | - El legajo debe ser único y no puede estar repetido<br>- Se valida que la fecha de contratación no sea futura<br>- El salario base debe ser mayor a cero<br>- Se asignan correctamente turno (para cajeros) o comisión (para asesores)<br>- El empleado se marca como activo por defecto<br>- La operación queda registrada en auditoría | Media | 2 |
| HU-13 | Como **cliente**, quiero recibir notificaciones automáticas del banco sobre movimientos de mis cuentas | - El sistema respeta la preferencia del cliente sobre recibir notificaciones<br>- Solo envía notificaciones a clientes con notificaciones activas<br>- Las notificaciones incluyen: tipo de operación, monto, fecha, saldo resultante<br>- Se notifica después de cada depósito, retiro, transferencia exitosa<br>- El cliente puede consultar si acepta o no notificaciones | Baja | 1 |
| HU-14 | Como **sistema auditor**, quiero consultar el historial de modificaciones en cuentas y clientes para auditar cambios y detectar anomalías | - Registra fecha exacta de cada cambio (con LocalDateTime)<br>- Identifica el usuario que realizó el cambio (legajo del empleado)<br>- Especifica qué se modificó (tipo de cambio)<br>- Se pueden listar todos los cambios o filtrar por entidad/fecha<br>- Los registros de auditoría no se pueden eliminar (son permanentes)<br>- Muestra un resumen legible de la auditoría | Baja | 2 |
| HU-15 | Como **cajero/asesor**, quiero ver un resumen de cualquier entidad (cliente, cuenta, empleado) para obtener información rápida | - Cada entidad (Cliente, Cuenta, Empleado) implementa `Consultable` con método `ObtenerResumen()`<br>- El resumen contiene los datos más relevantes en una línea de texto<br>- Se indica el tipo de entidad y estado (activa/inactiva)<br>- Se puede consultar el resumen en cualquier momento del ciclo de vida | Baja | 1 |
